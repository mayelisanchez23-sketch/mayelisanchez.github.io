Actividad 06 · Implementación IPSec VPN

1) Introducción

En esta actividad se realizó el montaje y configuración base de una VPN sitio-a-sitio con IPSec en Cisco Packet Tracer. El objetivo fue comprender el flujo general de implementación: validar conectividad entre sedes, definir el “tráfico interesante” mediante ACL y preparar la negociación inicial (ISAKMP/IKE) para el túnel VPN.

2) Desarrollo

2.1 Preparación del entorno
  • Se descargó e instaló Cisco Packet Tracer desde NetAcad.
  • Se verificó el acceso a recursos de Cisco Networking Academy para utilizar la herramienta.
2.2 Topología y direccionamiento

Se construyó una topología con dos sedes (LAN 192.168.1.0/24 y LAN 192.168.3.0/24) conectadas por un router “ISP” con enlaces públicos 209.165.100.0/24 y 209.165.200.0/24. El túnel IPSec se plantea entre los routers de borde.

2.3 Configuración inicial de interfaces y rutas

Se configuraron las interfaces LAN/WAN de cada router, y se agregaron rutas (incluyendo ruta por defecto hacia el ISP) para asegurar conectividad entre redes antes de aplicar cifrado.

2.4 Pruebas de conectividad (pre-VPN)

Se ejecutaron pruebas de conectividad mediante ping desde un host de una sede hacia la otra (ej. 192.168.3.10), confirmando que el enrutamiento funciona correctamente antes de iniciar la fase IPSec.

2.5 ACL para tráfico interesante

Se definió una ACL (access-list 100) para identificar el tráfico entre 192.168.1.0/24 y 192.168.3.0/24 como tráfico que debe viajar cifrado por el túnel. Esto es clave porque IPSec se activa al detectar tráfico que coincida con esa política.

2.6 Problema detectado (ISAKMP)

Al continuar con la fase de negociación, el entorno mostró que no se detecta ISAKMP, por lo que la configuración quedó incompleta. Esto puede ocurrir en Packet Tracer por limitaciones del modelo/IOS simulado o por diferencias entre comandos (ISAKMP vs IKEv2).

3) Material (evidencia)

Se adjunta evidencia en PDF con capturas de la instalación, topología, configuración inicial, pruebas de conectividad, ACL aplicada y el error relacionado con ISAKMP.

4) Reflexión

Esta práctica mostró que una VPN IPSec depende primero de una red estable: si el enrutamiento y la conectividad no son correctos, el túnel no puede levantarse. También dejó como aprendizaje que las herramientas de simulación pueden limitar la implementación, por lo que es necesario validar compatibilidad de comandos/modelos (por ejemplo, usar IKEv2 si aplica) o migrar a un entorno más completo para laboratorio cuando Packet Tracer no soporte ciertos componentes.
