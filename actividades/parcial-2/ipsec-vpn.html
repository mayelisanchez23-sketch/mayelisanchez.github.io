ACT_06 - Parcial 2

Implementacion IPSec VPN

# 1) Introduccion

En esta actividad se realizo el montaje y configuracion base de una VPN sitio-a-sitio con IPSec en Cisco Packet Tracer. El objetivo fue comprender el flujo general de implementacion: validar conectividad entre sedes, definir el "trafico interesante" mediante ACL y preparar la negociacion inicial (ISAKMP/IKE) para el tunel VPN.

# 2) Desarrollo

2.1 Preparacion del entorno

  • Se descargo e instalo Cisco Packet Tracer desde NetAcad.
  • Se verifico el acceso a recursos de Cisco Networking Academy para utilizar la herramienta.

2.2 Topologia y direccionamiento

Se construyo una topologia con dos sedes (LAN 192.168.1.0/24 y LAN 192.168.3.0/24) conectadas por un router "ISP" con enlaces publicos 209.165.100.0/24 y 209.165.200.0/24. El tunel IPSec se plantea entre los routers de borde.

2.3 Configuracion inicial de interfaces y rutas

Se configuraron las interfaces LAN/WAN de cada router, y se agregaron rutas (incluyendo ruta por defecto hacia el ISP) para asegurar conectividad entre redes antes de aplicar cifrado.

2.4 Pruebas de conectividad (pre-VPN)

Se ejecutaron pruebas de conectividad mediante ping desde un host de una sede hacia la otra (ej. 192.168.3.10), confirmando que el enrutamiento funciona correctamente antes de iniciar la fase IPSec.

2.5 ACL para trafico interesante

Se definio una ACL (access-list 100) para identificar el trafico entre 192.168.1.0/24 y 192.168.3.0/24 como trafico que debe viajar cifrado por el tunel. Esto es clave porque IPSec se activa al detectar trafico que coincida con esa politica.

2.6 Problema detectado (ISAKMP)

Al continuar con la fase de negociacion, el entorno mostro que no se detecta ISAKMP, por lo que la configuracion quedo incompleta. Esto puede ocurrir en Packet Tracer por limitaciones del modelo/IOS simulado o por diferencias entre comandos (ISAKMP vs IKEv2).

# 3) Material (evidencia)

Se adjunta evidencia en PDF con capturas de la instalacion, topologia, configuracion inicial, pruebas de conectividad, ACL aplicada y el error relacionado con ISAKMP.

# 4) Reflexion

Esta practica mostro que una VPN IPSec depende primero de una red estable: si el enrutamiento y la conectividad no son correctos, el tunel no puede levantarse. Tambien dejo como aprendizaje que las herramientas de simulacion pueden limitar la implementacion, por lo que es necesario validar compatibilidad de comandos/modelos (por ejemplo, usar IKEv2 si aplica) o migrar a un entorno mas completo para laboratorio cuando Packet Tracer no soporte ciertos componentes.
